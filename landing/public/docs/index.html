Documentation

Hatch spawn une preview isolée (app + DB + services) pour chaque pull request, avec une URL live commentée sur la PR. Cette page explique le format .hatch.yml, les variables disponibles, l'isolation et plusieurs exemples réels.

Pourquoi .hatch.yml ?

Vercel, Netlify, Cloudflare Pages excellent sur les apps stateless (frontend statique, fonctions edge). Dès que ton app a une base de données, Redis, un worker, des secrets — ça se complique. Tu te retrouves à :

  • Partager une DB staging entre tous les reviewers → pollution de données
  • Faire tourner l'app localement en installant la stack → reviewer non-tech perdu
  • Payer Vercel Pro + DB managées + Redis managé pour chaque PR → explose la facture

Hatch résout ça en lisant un fichier .hatch.yml à la racine du repo. Ce fichier déclare le stack complet nécessaire à ta PR — app, DB, cache, workers. Hatch spawn tout, isole chaque PR dans son propre réseau Docker, et nettoie à la fermeture.

TL;DR — si ton projet est une app stateless avec un seul Dockerfile, Hatch fonctionne sans .hatch.yml. Dès que t'as besoin d'une DB, .hatch.yml devient obligatoire pour déclarer les services.

Quickstart

En 3 étapes pour ajouter Hatch à un repo existant :

1. Installe la GitHub App

Sur github.com/apps/hatchpr, clique Install, sélectionne les repos où activer Hatch.

2. Ajoute un .hatch.yml à la racine

version: 1

services:
  web:
    build: .
    port: 3000
    expose: true
    env:
      DATABASE_URL: postgres://app:${DB_PASSWORD}@db:5432/app

3. Ouvre une PR

Hatch reçoit le webhook, build ton stack, route l'URL sur pr-<numéro>-<repo>.hatchpr.dev, et commente le lien sur la PR. Fermeture de la PR = cleanup total.

CLI @hatchpr/cli

Un binaire pour scaffolder ton .hatch.yml à partir de la stack détectée dans le repo (Node, Python, Rails, static, plus DB / Redis si présents).

# Install global
npm i -g @hatchpr/cli
hatch --help

# Ou one-shot, sans installer
npx @hatchpr/cli init

La seule commande aujourd'hui est hatch init — elle scanne le projet, propose un YAML, et l'écrit à la racine. Flags utiles :

  • --dry-run imprime le YAML sans écrire le fichier
  • --force écrase un .hatch.yml existant
  • --output <path> chemin de sortie (défaut .hatch.yml)
  • --verbose détaille la détection

Templates

Pour démarrer plus vite, fork un des templates prêts à l'emploi — Next.js, FastAPI, Django, Rails, static. Chacun embarque un .hatch.yml et un Dockerfile qui marchent en l'état.

Format .hatch.yml

Spec complète. Toutes les clés hors version et un services.* minimum sont optionnelles.

version: 1

services:
  <nom-du-service>:
    # Une seule des deux parmi build / image
    build: .                    # chemin du Dockerfile
    image: postgres:16-alpine    # OU image Docker pré-buildée

    port: 3000                  # port interne exposé
    expose: true                 # un seul service expose: true (routé via Traefik)

    env:
      KEY: valeur
      DATABASE_URL: postgres://user:${DB_PASSWORD}@db:5432/app

    depends_on: [db, redis]     # attente que ces services soient healthy

    healthcheck:
      cmd: pg_isready -U app
      interval_seconds: 3
      retries: 20

    limits:
      memory: 512m             # k/m/g, puissances de 1024
      cpu: 0.5                 # cœurs fractionnaires

seed:
  after: db                     # service qui doit être healthy avant exécution
  sql: ./seed/preview.sql        # script SQL appliqué une fois

Resource limits

Chaque service peut plafonner sa mémoire et son CPU. Les valeurs sont validées au parse : memory accepte les suffixes k, m, g (puissances de 1024), cpu est un nombre fractionnaire jusqu'à 16.

services:
  web:
    build: .
    limits:
      memory: 512m
      cpu: 0.5

Sans limits déclarés, les défauts appliqués sont 1 GiB de mémoire et 1 CPU par service. Mets des limites plus serrées sur tes workers / DB pour que les previews tiennent à plusieurs sur un même host.

Règles importantes

  • Un seul service doit avoir expose: true — c'est celui qui reçoit le trafic HTTPS public
  • Les services se joignent par leur nom court (db, redis, web) via DNS interne
  • port est auto-détecté depuis le Dockerfile si absent
  • Les valeurs env supportent la substitution de variables

Variables

Dans les valeurs env, tu peux utiliser :

VariableExempleDescription
${PR}42Numéro de la pull request
${SHA}a1b2c3dCommit SHA court (7 chars)
${REPO}itsaam-hatch-demo-nodeSlug du repo
${DB_PASSWORD}9f3a7e...Dérivé déterministe HMAC. Stable entre redeploys de la même PR, différent par PR.
${PREVIEW_HOST}pr-42-my-app.hatchpr.devHostname complet de la preview (sans protocole).
${PREVIEW_URL}https://pr-42-my-app.hatchpr.devURL complète HTTPS. Pratique pour NEXTAUTH_URL / APP_URL.
Secrets persistants — pour des clés API (Stripe test, SendGrid dev, etc.), utilise le secret store exposé sur le dashboard /app. Les valeurs sont chiffrées AES-256 côté serveur, scopées par repo, et injectées comme variables d'environnement runtime dans les services de chaque preview. Référence-les dans .hatch.yml via ${SECRET_NOM}.

Secret store

Géré depuis le dashboard /app. Liste, ajoute et supprime des secrets par repo. Stockés chiffrés AES-256 côté serveur, jamais en clair en base. Au runtime, chaque secret est exposé comme variable d'environnement aux services de la preview.

Référence un secret dans ton .hatch.yml avec le préfixe SECRET_ :

services:
  web:
    env:
      STRIPE_KEY: ${SECRET_STRIPE_TEST_KEY}
      SENDGRID_KEY: ${SECRET_SENDGRID_DEV}

Backup chiffré

Le dashboard expose un export .tar.gz.enc chiffré côté serveur, et un import qui relit ce ciphertext. Les secrets ne sont jamais déchiffrés côté client : le fichier circule en chiffré, et le serveur de destination peut le restaurer s'il partage la même clé maître.

Distinct des build secrets — un secret du store est injecté au runtime dans le container. Pour des secrets de build (token npm privé, etc.), passe par BuildKit et RUN --mount=type=secret ; ces secrets-là ne touchent jamais le runtime.

BuildKit

Hatch supporte BuildKit côté builder serveur. Deux directives utiles deviennent valides dans tes Dockerfiles :

Cache mounts

Les caches persistants entre builds accélèrent npm ci, pip install, bundle install, go mod download, etc.

# syntax=docker/dockerfile:1.4
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN --mount=type=cache,target=/root/.npm npm ci

Build secrets

Les secrets de build (token npm privé, clé GitHub pour go mod sur des repos privés, etc.) sont déclarés côté serveur par l'opérateur Hatch et injectés via la session gRPC BuildKit. Le secret n'apparaît jamais dans l'image finale.

# syntax=docker/dockerfile:1.4
RUN --mount=type=secret,id=npm_token \
    NPM_TOKEN=$(cat /run/secrets/npm_token) npm ci
Activation — la session BuildKit s'enclenche côté serveur via HATCH_BUILDKIT_SESSION=1. Si ton instance ne l'a pas activée, les directives --mount seront ignorées par le builder legacy.

Healthcheck

Hatch attend qu'un service soit healthy avant de démarrer ceux qui en dépendent (depends_on) et avant d'appliquer le seed. Sans healthcheck, il attend juste que le container soit running + 2s de grâce.

db:
  image: postgres:16-alpine
  healthcheck:
    cmd: pg_isready -U app -d app
    interval_seconds: 3   # défaut: 5s
    retries: 20             # défaut: 10

Le cmd est exécuté via sh -c dans le container. Timeout global pour devenir healthy : 90 secondes. Si dépassé, le deploy échoue.

Healthcheck HTTP post-deploy

Une fois les containers running, Hatch effectue une vérification HTTP supplémentaire sur le service exposé : il attend qu'au moins une requête soit servie sur le port public, avec un timeout de 60 secondes. Si rien ne répond dans ce délai, la preview bascule en failed au lieu de rester coincée en running avec un service muet.

Seed SQL

Seeds optionnels pour pré-remplir la DB avec un utilisateur de test, des données fixtures, etc. Exécuté une fois par PR après que le service seed.after soit healthy.

-- seed/preview.sql
CREATE TABLE IF NOT EXISTS users (
  id SERIAL PRIMARY KEY,
  email TEXT UNIQUE NOT NULL,
  password_hash TEXT NOT NULL
);

INSERT INTO users (email, password_hash) VALUES
  ('demo@example.com', '$2a$10$...')
ON CONFLICT (email) DO NOTHING;
Idempotent — ton seed peut être rejoué. Utilise CREATE TABLE IF NOT EXISTS, ON CONFLICT DO NOTHING, etc. Une erreur de seed n'empêche PAS le deploy (warning logué uniquement).

Isolation & cleanup

Pour chaque PR, Hatch crée :

  • Un réseau Docker dédié : hatch-net-<slug>-<pr> (bridge isolé)
  • Un container par service : hatch-pr-<slug>-<pr>-<service>
  • Des alias DNS internes : web, db, redis… pour que les services se causent sans connaître les noms complets

Seul le service expose: true est joignable de l'extérieur via Traefik + Let's Encrypt sur https://pr-<pr>-<slug>.hatchpr.dev.

Cleanup automatique

  • PR mergée ou fermée → suppression immédiate : containers + network (volumes DB inclus)
  • TTL d'inactivité → 7 jours par défaut, la preview hiberne et le bot commente. Reouvrir = redeploy.
  • Boot API → reconciliation : les containers orphelins sont détruits, les previews zombies marquées failed
Zéro pollution — chaque PR voit uniquement ses données. Deux reviewers qui inscrivent le même email sur deux PR différentes ne se marchent pas dessus.

Exemple : Node + Postgres

App Express qui expose /api/register et /api/login. Source du demo : github.com/itsaam/hatch-demo-node.

version: 1

services:
  web:
    build: .
    port: 3000
    expose: true
    env:
      DATABASE_URL: postgres://app:${DB_PASSWORD}@db:5432/app
      PORT: "3000"
    depends_on: [db]

  db:
    image: postgres:16-alpine
    env:
      POSTGRES_USER: app
      POSTGRES_PASSWORD: ${DB_PASSWORD}
      POSTGRES_DB: app
    healthcheck:
      cmd: pg_isready -U app
      interval_seconds: 3
      retries: 20

seed:
  after: db
  sql: ./seed/preview.sql

Exemple : Python (FastAPI/Django) + Postgres + Redis

version: 1

services:
  api:
    build: .
    port: 8000
    expose: true
    env:
      DATABASE_URL: postgresql://app:${DB_PASSWORD}@db:5432/app
      REDIS_URL: redis://redis:6379/0
      DEBUG: "true"
    depends_on: [db, redis]

  db:
    image: postgres:16-alpine
    env:
      POSTGRES_USER: app
      POSTGRES_PASSWORD: ${DB_PASSWORD}
      POSTGRES_DB: app
    healthcheck:
      cmd: pg_isready -U app

  redis:
    image: redis:7-alpine
    healthcheck:
      cmd: redis-cli ping

Exemple : Rails + Sidekiq

version: 1

services:
  web:
    build: .
    port: 3000
    expose: true
    env:
      DATABASE_URL: postgres://app:${DB_PASSWORD}@db:5432/app_pr_${PR}
      REDIS_URL: redis://redis:6379/0
      RAILS_ENV: production
      SECRET_KEY_BASE: ${DB_PASSWORD}
    depends_on: [db, redis]

  worker:
    build: .
    env:
      DATABASE_URL: postgres://app:${DB_PASSWORD}@db:5432/app_pr_${PR}
      REDIS_URL: redis://redis:6379/0
    depends_on: [db, redis]

  db:
    image: postgres:16-alpine
    env:
      POSTGRES_USER: app
      POSTGRES_PASSWORD: ${DB_PASSWORD}

  redis:
    image: redis:7-alpine

Le worker n'a pas de expose: true → il tourne en interne, pas accessible publiquement.

Exemple : Static (sans DB)

Pour un site statique ou une SPA, tu n'as pas besoin de .hatch.yml. Hatch détecte le Dockerfile à la racine, le build, et route directement.

# Dockerfile minimal suffit
FROM nginx:alpine
COPY ./dist /usr/share/nginx/html
EXPOSE 80

Dashboard & logs

Le dashboard /app liste tes previews actives et expose, pour chacune :

  • Le statut courant (queued, building, running, failed, hibernated)
  • Les logs de build et de runtime en streaming temps réel via SSE — pas de polling, les nouvelles lignes arrivent à la milliseconde
  • La gestion des secrets par repo (list / add / delete / backup chiffré)

Sécurité

  • Allowlist d'owners — les webhooks GitHub ne sont acceptés que pour les owners (user / org) listés dans HATCH_ALLOWED_OWNERS côté serveur. Toute PR d'un owner non listé est ignorée silencieusement.
  • PR de fork — refusées par défaut. Le code d'un fork ne peut pas builder sur ton infra, ni accéder au secret store.
  • Secrets en base — chiffrés AES-256, jamais loggés, jamais retournés en clair par l'API.

Durée de vie & TTL

  • Une preview active reste en ligne tant que la PR est ouverte
  • Sans activité (push, commit) pendant 7 jours, elle hiberne automatiquement : containers supprimés, DB purgée
  • Pour réveiller : push un commit sur la branche → Hatch redeploy à neuf
  • TTL configurable par l'admin du serveur (env PREVIEW_TTL_HOURS)

Troubleshooting

deploy failed: build ...

  • Vérifie que ton Dockerfile build en local : docker build .
  • Un .dockerignore manquant peut faire exploser la taille du contexte
  • Les logs complets sont visibles côté serveur (docker logs hatch-api-1)

seed failed (non-fatal)

  • Ton SQL a une syntax error ou référence une table inexistante
  • Rends ton seed idempotent avec CREATE TABLE IF NOT EXISTS et ON CONFLICT DO NOTHING
  • Le deploy continue quand même — seule la DB reste vide

Le service ne peut pas joindre la DB

  • Vérifie que tu utilises le nom court du service dans l'URL (db, pas hatch-pr-xxx-db)
  • Ajoute depends_on: [db] pour garantir l'ordre de démarrage
  • La DB doit avoir un healthcheck pour que depends_on attende vraiment qu'elle soit prête

Certificat SSL invalide pendant ~30s

Let's Encrypt émet le cert à la volée. Lors du premier accès à une nouvelle PR, attends 30s et force-refresh.

Limites actuelles

Hatch est en beta. Ce qui n'est pas encore supporté :

  • Docker 29+ avec containerd snapshotter par défaut — le builder utilisé par Hatch ne fonctionne pas sur cette configuration. Utilise un Docker antérieur (< 29) ou désactive le containerd snapshotter pour l'instant.
  • Pas de volumes persistants — la DB d'une preview est toujours repartie de zéro au redeploy (et c'est souvent ce qu'on veut)
  • Pas de multi-region — une seule infra host pour l'instant
